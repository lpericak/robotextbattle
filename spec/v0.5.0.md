# v0.5.0 Spec

## 1. Quick Rematch

After a fight ends, offer "Rematch" alongside "Continue" (and "Battle Log"
if that exists). Rematch starts a new fight against the same enemy without
returning to the menu.

### Behavior

- Appears on the victory/defeat/surrender results screen
- Choices: `Rematch`, `Continue`
- Rematch: reset player HP/energy, create a fresh battle against the same
  enemy, increment fight number. No save until returning to menu.
- Continue: existing behavior (return to main menu, save)

### Implementation

In `battleScreen()`, replace the final `promptContinue()` with a
`promptChoice` offering Rematch/Continue. If Rematch, loop back to the
top of the battle function. Wrap the existing battle logic in a loop
that breaks on Continue.

The function signature stays the same — the caller doesn't need to know
about rematches.

### Save timing

Do **not** save between rematches. Save happens when the player returns
to the main menu (existing behavior). This keeps the save logic simple
and avoids partial-session saves.

## 2. Sound Effects

Add basic Web Audio API sound effects. No audio files — generate tones
programmatically.

### Sounds

| Event | Sound | Duration |
|-------|-------|----------|
| Hit | Short noise burst (white noise, 50ms) | 50ms |
| Miss | Quick descending tone (400→200 Hz) | 100ms |
| Victory | Rising three-note arpeggio (C5→E5→G5) | 300ms |
| Defeat | Descending two-note (G4→C4) | 300ms |
| Level up | Quick ascending scale (C5→E5→G5→C6) | 400ms |
| Buy item | Cash register "cha-ching" (two quick high tones) | 150ms |
| Menu select | Soft click (very short tone, 800 Hz) | 30ms |

### Architecture

Create `src/ui/sound.ts` with:

```typescript
interface SoundPlayer {
  hit(): void;
  miss(): void;
  victory(): void;
  defeat(): void;
  levelUp(): void;
  buy(): void;
  click(): void;
  setEnabled(on: boolean): void;
}
```

Use a single shared `AudioContext`, created lazily on first user
interaction (required by browser autoplay policy). All sounds are
generated with `OscillatorNode` and `GainNode` — no audio files.

### Integration

- Pass `SoundPlayer` to screen functions alongside `Terminal`
- Battle screen: `hit()` on damage, `miss()` on miss, `victory()`/`defeat()` on end
- Shop: `buy()` on purchase
- State: `levelUp()` on level up
- Terminal: `click()` on choice selection

### Settings integration

Add a "Sound: On/Off" toggle to the Settings screen. Persist the
preference in localStorage under `robot-battle-sound`. Default: On.

## 3. Save Slots

Replace single-save with 3 named slots so siblings can each have
their own game.

### Title screen flow

```
ROBOT BATTLE v0.5.0

1. [Slot 1] Oliver Lv.5 — $340
2. [Slot 2] Empty
3. [Slot 3] Lucas Lv.2 — $110

Select a slot to play, or start a new game in an empty slot.
```

### Storage

- Keys: `robot-battle-save-1`, `robot-battle-save-2`, `robot-battle-save-3`
- Same JSON format per slot: `{ version: 1, player: Robot }`
- Settings (sound, game mode) are stored per-slot inside the save:
  add an optional `settings` field to the save format
- Bump save version to 2. Migrate version-1 saves into slot 1 on
  first load (backwards compatible)

### Title screen behavior

- Show all 3 slots with status (name + level + money, or "Empty")
- Click an occupied slot → load it, go to main menu
- Click an empty slot → "Name your robot" prompt, save to that slot
- Add a "Delete" option per occupied slot (with confirmation)

### Implementation

- Rename `SAVE_KEY` to `SAVE_KEY_PREFIX = "robot-battle-save-"`
- New functions: `listSlots(storage)`, `loadSlot(storage, slot)`,
  `saveSlot(storage, slot, player)`, `deleteSlot(storage, slot)`
- `hasSave` becomes `hasAnySlot`
- Game screen tracks `activeSlot: number` and passes it through
  to save calls
- Migration: on load, if `robot-battle-save` exists (old format),
  move it to slot 1 and delete the old key

## 4. Minimal Combat Animations

CSS-only animations for combat feedback. No JavaScript animation
libraries.

### Effects

| Event | Animation | CSS |
|-------|-----------|-----|
| Hit taken | Panel shakes horizontally | `@keyframes shake` — translate ±4px, 200ms |
| Miss | Brief flash/dim of attacker panel | `@keyframes flash` — opacity 0.5→1, 150ms |
| Defeat | Losing panel fades out | `@keyframes fadeOut` — opacity 1→0.3, 500ms |
| Victory | Winning panel pulses green | `@keyframes pulse` — border-color bright→dim, 500ms x2 |
| Level up | Gold flash on stats bar | `@keyframes goldFlash` — background-color gold→transparent, 400ms |

### Implementation

Add keyframes to `terminal.css`. Apply via CSS classes added/removed
by the battle screen:

- `printBattleStatus` adds animation classes based on turn log content
- After printing turn resolution, check log for "hits for" → add
  `.shake` to the defender's panel
- Check for "misses" → add `.flash` to attacker's panel
- On battle end, add `.fadeOut` to loser, `.pulse` to winner

Classes are removed on next render (terminal.clear removes all DOM).

### What NOT to animate

- Shop interactions (keep snappy)
- Menu navigation (keep instant)
- Auto-battle turns (too fast, animations would overlap)

## 5. Bug Fix: Ammo Stacking

### Problem

Shotgun Shells are typed as `gear` but function as consumable ammo.
Two bugs result:

1. **Can't buy multiples**: `canBuy()` rejects duplicate gear
   (`"You already have this gear equipped"`), so you can only hold 1 shell
2. **Each shell wastes a full inventory slot**: 3 shells = 3 of 4 slots

### Root cause

Shotgun Shell is in the `gear` section of `items.json` because it
provides a passive capability (enabling Sawed-off Shotgun). But it's
consumed on use (battle.ts line 113-117 removes the requirement item
from inventory after firing). This makes it hybrid gear/consumable,
which the current type system doesn't support.

### Solution: `stackable` flag on gear

Add an optional `stackable: true` field to gear items. Stackable gear:

- **Bypasses the duplicate check** in `canBuy()` — you can buy multiples
- **Counts as 1 inventory slot** regardless of quantity — the inventory
  display shows "Shotgun Shell x3" instead of 3 separate entries
- **Stays as gear type** — it still appears in gear lists, provides
  the `hasItem` check for weapon requirements, and gets consumed by
  battle.ts requirement logic

### Type changes

```typescript
// In Gear interface, add:
stackable: boolean;  // default false
```

### Engine changes

**shop.ts `canBuy()`**: Skip the "already have this gear" check when
`item.stackable === true`.

**shop.ts `canBuy()` inventory size check**: Count stackable gear
groups as 1 slot. Add a helper:

```typescript
function countInventorySlots(player: Robot): number {
  const stacked = new Set<string>();
  let count = 0;
  for (const item of player.inventory) {
    if (item.itemType === "gear" && (item as Gear).stackable) {
      if (!stacked.has(item.name)) {
        stacked.add(item.name);
        count++;
      }
    } else {
      count++;
    }
  }
  return count;
}
```

Replace `player.inventory.length >= player.inventorySize` with
`countInventorySlots(player) >= player.inventorySize`.

**data.ts `loadGear()`**: Read `stackable` field, default `false`.

### UI changes

**shop.ts buy menu**: Show current stack count for stackable items:
"Shotgun Shell x3 — $30" instead of plain "Shotgun Shell — $30".

**inspect.ts inventory display**: Group stackable items:
"Shotgun Shell x3" on one line, not three separate lines.

**shop.ts sell menu**: Sell one at a time from the stack. Show
"Sell Shotgun Shell (have 3) for $15".

### Data changes

```json
"Shotgun Shell": {
  "level": 5,
  "moneyCost": 30,
  "stackable": true,
  "description": "Allows you to fire a Shotgun weapon once. Consumed when fired."
}
```
