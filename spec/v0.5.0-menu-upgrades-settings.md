## Summary

Three changes in one release:

1. **Shop tab bar** (already implemented) вҖ” Buy/Sell/Inventory/Back as
   persistent tabs at the top of the shop screen

## 1. Shop Tab Bar (done)

Shop navigation is now a tab bar rendered at the top of every shop view.
All four tabs (Buy, Sell, Inventory, Back) are always visible. The active
tab is highlighted with a green background. Back is right-aligned in yellow.

Implementation details:
- `Choice` interface gained `group` and `active` fields
- Grid layout in `promptChoice` splits choices into a `.tab-bar` row
  (for `group: "tab"` choices) and a `.card-grid` below
- `promptChoice` accepts optional `contentHTML` (4th param) inserted
  between tabs and grid вҖ” used by the Inventory tab
- CSS classes: `.tab-bar`, `.tab`, `.tab-active`, `.tab-back`

## 2. Remove "Inspect Robot"

The main menu currently has 4 options: Fight, Shop, Inspect Robot, Quit.

Remove "Inspect Robot". The shop's Inventory tab shows the same
information (stats, weapons, gear, consumables). Having both is

## 3. Upgrades Screen

Permanent, expensive buffs that persist across fights. These are not
inventory items вҖ” they are permanent unlocks stored on the Robot object.

### Upgrade definitions

| Name | Cost | Effect | Description |
|------|------|--------|-------------|
| Inventory 5 | $5,000 | inventorySize вҶ’ 5 | Add a 5th inventory slot |
| Inventory 6 | $20,000 | inventorySize вҶ’ 6 | Add a 6th inventory slot |

No level requirements. The cost is the gate.

Inventory 6 requires Inventory 5 to be purchased first (sequential).

### Data model

Add an `upgrades` field to the `Robot` interface:

```typescript
interface Robot {
  // ... existing fields ...
  upgrades: string[];  // list of purchased upgrade IDs
}
```

`upgrades` starts as `[]` for new players. Each upgrade ID is a string
like `"inventory-5"`, `"inventory-6"`. The upgrade definitions live in
a new engine file, not in JSON вҖ” they're hardcoded since they affect
game mechanics directly.

### Engine: `src/engine/upgrades.ts`

```typescript
interface UpgradeDef {
  id: string;
  name: string;
  cost: number;
  description: string;
  requires: string | null;  // prerequisite upgrade ID
  apply(player: Robot): void;
  canApply(player: Robot): boolean;
}
```

All upgrade definitions in a single `UPGRADES` array. Functions:

- `listUpgrades()` вҖ” returns all upgrade defs
- `canBuyUpgrade(player, upgrade)` вҖ” checks money, prerequisites,
  and not already purchased
- `buyUpgrade(player, upgrade)` вҖ” deducts money, adds to
  `player.upgrades`, applies effect
- `applyAllUpgrades(player)` вҖ” re-applies all purchased upgrades
  (called on save load to ensure effects are active)

### Save compatibility

Adding `upgrades: string[]` to Robot is a forward-compatible change.
Old saves without `upgrades` should default to `[]` on load. Add a
migration step in `loadGame()`:

```typescript
if (!data.player.upgrades) data.player.upgrades = [];
```

The save version number stays at 1 вҖ” this is a soft migration, not a
breaking change.

## 4. Settings Screen

Game settings stored on the Robot object (so they persist with the save).

### Settings data model

Add a `settings` field to the `Robot` interface:

```typescript
interface Robot {
  // ... existing fields ...
  settings: {
    mode: "oliver" | "lucas";
  };
}
```

Defaults to `{ mode: "oliver" }` for new players. Old saves without
`settings` get the default on load (same soft migration as upgrades).

### Mode behavior

- **Oliver mode** (default): normal gameplay, all energy costs apply
- **Lucas mode**: all weapon energy costs are treated as 0

The energy cost override is applied in the battle engine wherever
`weapon.energyCost` is read. Add a helper:

```typescript
function getWeaponEnergyCost(weapon: Weapon, player: Robot): number {
  if (player.settings?.mode === "lucas") return 0;
  return weapon.energyCost;
}
```

This affects:
- `planAttack()` вҖ” energy check when selecting weapons
- `resolveTurn()` вҖ” energy deduction after attacking
- AI weapon selection вҖ” `aiPlanAction()` energy calculations
- Shop item summary вҖ” show "0 en" in Lucas mode
- Battle UI вҖ” energy cost display

### Hover tooltips

Add a general-purpose tooltip system to the terminal for future reuse.

#### Implementation

CSS-only tooltips using `data-tooltip` attribute + `::after` pseudo-element:

```css
[data-tooltip] {
  position: relative;
}

[data-tooltip]:hover::after {
  content: attr(data-tooltip);
  position: absolute;
  bottom: 100%;
  left: 50%;
  transform: translateX(-50%);
  background: #0a0a0a;
  border: 1px solid #33ff33;
  border-radius: 4px;
  padding: 4px 8px;
  font-size: 13px;
  white-space: nowrap;
  z-index: 50;
  pointer-events: none;
  color: #33ff33;
}
```

No JavaScript needed. Any element with `data-tooltip="text"` gets a
tooltip on hover. Used on the Settings screen for mode buttons:

```html
<div class="btn" data-tooltip="Normal gameplay вҖ” energy costs apply">Oliver</div>
<div class="btn" data-tooltip="All energy costs are 0">Lucas</div>
```

The tooltip infrastructure is generic and can be reused anywhere вҖ” shop
items, upgrade cards, battle actions, etc.
